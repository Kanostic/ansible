# CNS Ansible + Terraform Cloud EC2 deployment

Use one of these playbooks in your AAP Job Template:
- terraform-aws-gn-base-virtual-appliance.yml

How it works (high level)
- Provisions EC2 via Terraform Cloud (TFC). SSH keys are generated by Terraform and persisted in state.
- Extracts instance IDs from TFC logs, discovers public/private IPs.
- SSH into each instance and runs command_to_run (global or per-instance). It exports AWS credentials in-session before executing your command.
- Optional S3 upload role can be enabled by setting s3_bucket_name/prefix.

Paste this into AAP “Extra Vars”
```yaml
# Global command_to_run (applied to all instances unless overridden per instance)
command_to_run: |

# Global user_data (applied to all instances unless overridden per instance)
user_data: |
  #!/bin/bash

# Instances list
instances:
  - aws_region:
    vpc_id:
    aws_account_id: ''
    command_to_run: |  # This overrides the global command_to_run for this instance
      echo "hello from ssh!"

  - aws_region:
    vpc_id:
    aws_account_id: ''

  # Add more instances as needed...
```

Example Extra Vars (two instances)
```yaml
environment: dev
create_key_pair: true
instance_type: m7i.large
root_volume_size: 250

# Runs for all instances unless overridden
command_to_run: |
  echo "hello from global"; uname -a

# Optional, used by cloud-init on boot (per-instance value takes precedence)
user_data: |
  #!/bin/bash
  echo "bootstrapping $(hostname)" > /var/tmp/bootstrap.txt

# Optional S3 upload (enables IAM instance profile on the instance)
s3_bucket_name: mi-cns-orgshared-bucket-38asdh2
s3_bucket_prefix: connectivity_results/

instances:
  - aws_region: us-east-1
    vpc_id: vpc-0123456789abcdef0
    aws_account_id: '111122223333'
    command_to_run: |
      echo "hello from us-east-1"; curl -s ifconfig.me || true

  - aws_region: us-west-2
    vpc_id: vpc-0fedcba9876543210
    aws_account_id: '444455556666'
```

Notes
- Per-instance command_to_run and user_data override global values when provided.
- SSH target is Public IP if present, else Private IP. Ensure AAP runner can reach the target on port 22.
- Security group must allow SSH from your AAP runner.
- AMI is auto-selected by region unless ami_id is provided.
- Valid environment values: dev, test, stg, perf, prod.
- To destroy, set tfc_state: destroy (and ensure tfc_allow_destroy as appropriate).

## Redeploying Single Instances Safely

To redeploy a specific instance without affecting others or re-running bootstrap scripts, use this approach:

### Scenario 1: Redeploy One Instance (Patch/Reboot)

Use `terraform_targets` to target specific modules and set `skip_ssh_keygen: true` to prevent SSH key regeneration:

```yaml
instances:
  - aws_region: us-east-2
    vpc_id: vpc-039f134d0a2b50b7d
    aws_account_id: '256177873118'
    user_data: >

  - aws_region: ap-south-2
    vpc_id: vpc-0496ef79c9a35dd44
    aws_account_id: '256177873118'
    user_data: >

tfc_state: apply
skip_ssh_keygen: true          # Don't regenerate SSH key
skip_user_data: true            # Don't re-run user_data scripts
terraform_targets:
  - module.ec2-1                # Redeploy only the first instance
```

**What this does:**
- Targets only `module.ec2-1` (first instance in the list)
- Skips SSH key regeneration
- Excludes user_data from the Terraform config (existing instance is untouched)
- Other instances remain completely unaffected

### Scenario 2: Add New Instances to Existing Deployment

Keep all existing instances in the list and add new ones. The system will only create new modules and won't touch existing ones:

```yaml
instances:
  # Existing instances (unchanged)
  - aws_region: us-east-2
    vpc_id: vpc-039f134d0a2b50b7d
    aws_account_id: '256177873118'
    user_data: >

  - aws_region: ap-south-2
    vpc_id: vpc-0496ef79c9a35dd44
    aws_account_id: '256177873118'
    user_data: >

  # New instance (will be created as module.ec2-3)
  - aws_region: eu-west-1
    vpc_id: vpc-0abc1234def5678
    aws_account_id: '999988887777'
    user_data: >
      #!/bin/bash
      echo "New instance bootstrap"

tfc_state: apply
create_key_pair: true           # SSH key generated (can use same key for all)
terraform_targets: []          # Empty or omitted = deploy everything
```

**What this does:**
- Creates new `module.ec2-3` for the new instance
- Existing instances are not re-targeted (safe)
- User_data runs only on the new instance
- SSH key is generated once and shared across all instances

### Scenario 3: Only Update Configuration (No Terraform Changes)

If you only need to re-run the SSH commands but not modify infrastructure:

```yaml
instances:
  - aws_region: us-east-2
    vpc_id: vpc-039f134d0a2b50b7d
    aws_account_id: '256177873118'
    command_to_run: |
      echo "Running updated commands"

tfc_state: apply
terraform_targets:
  - skip-terraform-entirely      # Special target to skip Terraform entirely
```

### Variables Reference for Safe Redeployment

| Variable | Type | Default | Purpose |
|----------|------|---------|---------|
| `skip_ssh_keygen` | bool | false | Set to `true` to prevent SSH key regeneration during redeploy |
| `skip_user_data` | bool | false | Set to `true` to exclude user_data from Terraform (won't re-run bootstrap) |
| `terraform_targets` | list | [] | Specific modules to deploy (e.g., `["module.ec2-1"]`). Empty list = all instances |
| `tfc_state` | string | apply | Set to `apply` for deployment, `plan` for preview, `destroy` to remove |
| `create_key_pair` | bool | true | Generate SSH key pair. Disable only if reusing existing keys |

### Best Practices

1. **Always keep the full instances list** - Don't remove old instances when adding new ones
2. **Use terraform_targets for targeted changes** - Prevents accidental modifications to other instances
3. **Test with tfc_state: plan first** - Preview what changes will be made before applying
4. **Keep user_data empty after first deploy** - Or set `skip_user_data: true` on redeploys
5. **Document your deployment** - Save the vars you used for each deployment phase

## Safely Deleting Single Instances

**⚠️ CRITICAL:** Do NOT use `terraform_targets` with `tfc_state: destroy`. It can destroy all instances unexpectedly.

### Safe Method: Remove from List

```yaml
# STEP 1: Preview with plan
instances:
  - aws_region: us-east-2
    vpc_id: vpc-039f134d0a2b50b7d
    aws_account_id: '256177873118'
    user_data: >

  - aws_region: ap-south-2
    vpc_id: vpc-0496ef79c9a35dd44
    aws_account_id: '256177873118'
    user_data: >

  # af-south-1 REMOVED (not in list)

tfc_state: plan  # Preview first!
```

Review the plan output to confirm only module.ec2-3 will be destroyed.

```yaml
# STEP 2: Apply destruction
# Same config as above, just change to:
tfc_state: destroy
```

Only the missing instance will be destroyed; others are preserved.

## SSH Key Management

SSH keys are now managed entirely by Terraform using a **shared key architecture**. This ensures that redeploying a single instance does not regenerate the SSH key for any instance.

### How It Works

1. **Single `tls_private_key` resource** - One SSH key is generated for the entire deployment
2. **Per-region `aws_key_pair` resources** - The same key is registered in each AWS region (with `lifecycle { ignore_changes = [public_key] }`)
3. **Shared across all modules** - Each EC2 instance references the shared key pair

### Key Benefits

- **Redeploying `module.ec2-3` won't regenerate the SSH key** - The `tls_private_key.shared` resource is not targeted
- **All instances use the same key** - Simplifies access management
- **Key persists in Terraform state** - Available via `terraform output`

### Retrieving the Private Key

After deployment, retrieve the private key from Terraform Cloud:

```bash
# From TFC workspace outputs
terraform output -raw shared_private_key_pem > ~/.ssh/ec2-key.pem
chmod 600 ~/.ssh/ec2-key.pem
```

### Using an Existing Key

If you already have an SSH key and don't want Terraform to generate one:

```yaml
instances:
  - aws_region: us-east-2
    vpc_id: vpc-039f134d0a2b50b7d
    aws_account_id: '256177873118'
    existing_key_name: my-existing-key-pair  # Use existing AWS key pair

# Or set globally for all instances:
existing_key_name: my-existing-key-pair
```

### Safe Redeployment Example

```yaml
instances:
  - aws_region: us-east-2
    vpc_id: vpc-039f134d0a2b50b7d
    aws_account_id: '256177873118'
    user_data: >

  - aws_region: ap-south-2
    vpc_id: vpc-0496ef79c9a35dd44
    aws_account_id: '256177873118'
    user_data: >

  - aws_region: af-south-1
    vpc_id: vpc-0166a5d499bac3cce
    aws_account_id: '256177873118'
    user_data: >
      #!/bin/bash
      # New user_data for ec2-3 only

tfc_state: apply
terraform_targets:
  - module.ec2-3  # Only redeploy this instance
```

**What happens:**
- ✅ `module.ec2-3` is replaced with new user_data
- ✅ SSH key remains unchanged (not in target list)
- ✅ Other instances (`ec2-1`, `ec2-2`) are untouched
